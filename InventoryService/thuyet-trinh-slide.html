22.7.3 // Architectural Pattern

Two-Phase Commit = all or nothing

“…ensure that updates to inventory, credit card, and delivery order are atomic, so either all committed or all aborted.”

Vì sao tách reserve / commit

Khi đặt hàng ta chưa trừ kho, chỉ giữ chỗ (prepare). Coordinator đợi mọi bên “vote yes”. Tất cả OK → commit. Một bên từ chối → abort.

Vai trò của Inventory

Participant thụ động: chỉ vote (reserve OK/fail) và thực thi lệnh commit/abort của Coordinator. Rollback rất sạch vì reserve chưa từng đụng quantity.

Implementation // Chống race condition

Guard nằm trong WHERE = atomic

Thay vì đọc-rồi-ghi (có khe hở cho 2 đơn giành 1 món), điều kiện kiểm tra nằm ngay trong mệnh đề WHERE — một câu lệnh atomic duy nhất.

UPDATE inventory
   SET quantityReserved = quantityReserved + @amount
 WHERE itemId = @itemId
   AND (quantity - quantityReserved) >= @amount   -- guard

Không đủ hàng → changes === 0 → ném lỗi 409 (= vote abort). Đây là lý do chọn better-sqlite3: transaction đồng bộ, atomic sạch. Xem inventory.repository.js.

Chạy thử

● Luồng chính · reserve → commit

POST /inventory/1001/reserve  {"amount":2}
POST /inventory/1001/commit   {"amount":2}
→ quantity −2, reserved → 0  (S9→S10)

● Hủy đơn · reserve → abort

POST /inventory/1001/reserve  {"amount":2}
POST /inventory/1001/abort    {"amount":2}
→ quantity KHÔNG đổi (rollback)

● Hết hàng · Out of Stock

POST /inventory/1003/reserve  {"amount":99}   → 409 Out of stock

Kết luận

Inventory Service là một participant thụ động trong SOA: expose đúng interface, đóng vai prepare / commit / abort trong two-phase commit, và giữ nhất quán tồn kho bằng các thao tác atomic.
